This project was my first attempt at building a system where multiple components had to work together rather than treating each feature independently. Instead of focusing only on visuals, I wanted to build a structure where rendering, physics, input, camera logic, resource loading, and world organisation were coordinated through a single architecture.

To manage this complexity, I organised the project into modular responsibilities. Rendering, vehicle behaviour, physics simulation, camera control, timing, input handling, and world construction were separated into their own systems. This separation became necessary as the project grew, because the main challenge shifted from simply making features work to making the system understandable enough to extend and debug.

The vehicle system itself required careful iteration. While basic movement is straightforward to implement, making the behaviour feel controlled and readable required balancing acceleration, steering response, braking behaviour, and physical stability. This forced me to think about the relationship between simulation and perception — how small changes in physics values could significantly affect how the movement felt to the user.

One of the main challenges was managing interactions between systems. Adjustments in physics could affect controls, while camera changes could alter how motion was perceived. Solving this required repeated testing and refinement rather than one-time implementation. I learned that building complex systems is less about adding features and more about coordinating how existing parts interact.

As the project expanded, scene organisation also became critical. I separated terrain generation, respawn logic, vehicle setup, rendering flow, and input updates into structured components. This reinforced an important lesson: architecture is not something added after a project grows large — it becomes necessary as soon as complexity begins to accumulate.

This project changed how I think about programming. Earlier work focused on isolated features, but this experience required thinking about dependencies, update flow, and how data moves between systems. I began focusing more on how to maintain flexibility while increasing technical complexity.

What I value most about this project is not the visual result, but what it taught me about structure. It reinforced the idea that complex systems require organisation, iteration, and the ability to reduce complexity into smaller interacting parts. This connection between structure in mathematics and structure in software is a major reason I continue to pursue computer science.
